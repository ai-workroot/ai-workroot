# AI Workroot 0.9.531 Release Notes

## Scope

0.9.531 is the Agent Protocol and Task Continuity foundation release.

This release builds on the 0.9.530 Clean Workroot architecture reset. It adds a stable Agent/LLM-to-Workroot control loop for semantic task continuity while keeping Workroot runtime state out of the user-selected directory.

## Highlights

- Added the Workroot Agent Protocol response envelope for `sync` and `commit`.
- Added the Workroot private packet renderer for LLM-facing, private, compact protocol guidance.
- Added read-mostly startup context that does not create Task, TaskRun, lease, or event facts by itself.
- Added reliable commit batching with SQLite `BEGIN IMMEDIATE`, semantic idempotency hashes, stored response replay, and conflict detection.
- Added lease safety for durable commit attempts, including resync-required responses when state versions changed.
- Added task continuity projections for Task, TaskRun, TaskSummary, TaskItem, and Handoff.
- Added inbox/temporary task support and promotion through Task metadata updates.
- Added early Asset and Decision capture into durable projections, context candidates, relationships, and local indexing.
- Added rebuildable runtime read views under per-Workroot semantic directories such as `tasks/`, `handoffs/`, `assets/`, `relationships/`, `indexes/`, `context/`, and `diagnostics/`.
- Added live and integration coverage for Agent protocol loops, task owner continuity, runtime views, and private packet behavior.

## Guardrails

- External protocol actions remain limited to `sync` and `commit`.
- `sync` is read-mostly and must not create Task or TaskRun facts.
- Durable facts enter through `commit` only.
- Quick work does not create durable Workroot facts.
- A `commit(intent, persistence=quick)` attempt records the batch outcome for idempotency but does not project a durable event, Task, or TaskRun.
- Strong durable user intent must not be discarded solely because an Agent or model reports a quick work signal.
- Completed TaskRun state does not force a Handoff. Incomplete TaskRun state asks for continuation before stopping.
- Task completion is represented as `Task.status=closed` with a close reason, not as a `completed` Task status.
- Task remains the single work abstraction. Temporary work, inbox work, long-running work, and parent/child relationships are expressed through Task metadata and relationships, not through Project, Initiative, or SubTask concepts.
- Runtime read views are rebuildable diagnostics and query views. SQLite remains the canonical fact store at `cache/workroot.sqlite`.
- User directories remain user asset space. Workroot runtime folders, indexes, logs, and internal state are not written there by default.

## Known Limitations

- Layered context recall and disclosure now has an initial WorkSignal/lease-driven skeleton, but semantic ranking and evidence drill-down remain active iteration areas.
- Focus resolution uses stable protocol semantics for work boundaries. Owner disambiguation still uses compact local matching and needs further semantic ranking hardening.
- Asset and Decision capture are early continuity capabilities, not a complete asset/decision lifecycle and recall system.
- Focus, projection, and context builder modules remain future split points. This release keeps behavior stable rather than doing a large file-structure refactor.
- MCP and Skill-first adapters are future entry surfaces. The stable protocol contract is the priority in this release.
- Unexpected projection/storage exceptions still surface for tests and diagnostics instead of always producing a terminal rejected protocol response.
